# Codebase Knowledge Layer — Open Questions

These questions need resolution before or during implementation of the `system-explainer` skill and the broader codebase knowledge layer.

---

## 1. ~~Knowledge Cache Generation — Who Triggers It?~~ ✅ RESOLVED

**Decision**: Two-tier approach:
- **Baseline overviews**: Engineer-generated via Claude Code `/init` → stored in `.opencode/context/repos/`
- **Knowledge cache docs**: Auto-generated by `system-explainer` on TPM query → stored in Obsidian `50-Systems/`
- Engineers own the baseline; agent + TPM questions build up the knowledge cache

---

## 2. MVP Phasing — Where Does This Fit?

**Question**: Which implementation phase does the codebase knowledge layer belong to?

**Options**:
- Phase 3 (alongside Atlassian integration)
- New Phase 5 (after Polish & Distribution)
- Post-MVP

**Consideration**: This is high-value for TPMs but doesn't depend on Atlassian MCP. Could be Phase 2.5 (after Obsidian Skills are set up) or a parallel track.

---

## 3. Scope of Repos — Which Repos to Include?

**Question**: Whitelist of 2-3 core repos, or all team repos?

**Recommendation**: Start with 2-3 core services that TPMs ask about most. Expand on demand.

**Action needed**: Identify the top 2-3 repos for the pilot.

---

## 4. Knowledge Freshness — Auto-Refresh or On-Demand?

**Question**: Should cached system knowledge docs auto-refresh on a schedule, or only when a user says "refresh"?

**Options**:
- **On-demand only** (MVP): User explicitly asks to refresh
- **Scheduled**: Cron job re-generates docs weekly
- **Change-triggered**: Watch git log for changes since `generated-date`, flag stale docs

**Leaning**: On-demand for MVP. Add staleness detection later (check git log date vs generated-date).

---

## 5. Engineer-in-the-Loop — Review Auto-Generated Docs? (Partially Resolved)

**Partially resolved**: Baseline overviews (CLAUDE.md) are engineer-generated, so they're inherently reviewed. Knowledge cache docs (auto-generated by system-explainer) are still LLM-generated.

**Remaining question**: Should knowledge cache docs have a review workflow?

**Leaning**: Use `status: draft` for auto-generated, `status: reviewed` after engineer sign-off. No blocking workflow — TPMs can use draft docs immediately. Baseline overview provides a trustworthy foundation regardless.

---

## 6. Repo Maintenance — Clone Freshness & Corruption

**Question**: How often to `git pull`? What if a clone gets corrupted?

**Options for pull frequency**:
- Manual (TPM or engineer runs a refresh command)
- Daily cron job
- On-demand before each system-explainer query

**Corruption handling**: If clone is corrupted, delete and re-clone. Could add a `repo-health-check` utility.

**Leaning**: Daily cron pull for MVP. Add health check later.

---

## 7. Canvas Generation — Should system-explainer Create Architecture Views?

**Question**: Should `system-explainer` generate `.canvas` files showing service architecture?

**Consideration**: Canvas files are great for visual architecture maps, but auto-generating them adds complexity. Mermaid diagrams in markdown are simpler and already supported.

**Leaning**: Mermaid diagrams in markdown for MVP. Canvas generation as a future enhancement.
